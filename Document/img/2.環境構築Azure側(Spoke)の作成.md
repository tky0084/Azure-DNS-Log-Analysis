### 仮想マシンの作製

Azure Portalで「仮想マシン」を検索し、「作成」を押下する

![alt text](<スクリーンショット 2026-04-13 084931.png>)

<br>

基本タブにて、次の通りに入力する

| 項目 | 入力方法 |
| -- | -- |
| サブスクリプション | Spoke環境のリソースグループがあるサブスクリプション |
| リソース グループ | Spokeの仮想ネットワークがあるリソースグループ |
| 仮想マシン名 | 任意の仮想マシン名 |
| リージョン | (Asia Pacific) Japan East |
| 可用性オプション | インフラストラクチャ冗長は必要ありません |
| セキュリティの種類 | Standard |
| イメージ | Windows Server 2025 Datacenter: Azure Edition - x64 Gen2 |
| サイズ | デフォルト |
| ユーザー名 | 仮想マシンの管理者アカウント名 |
| パスワード | 管理者アカウントのパスワード |
| パブリック受信ポート | なし |

![alt text](<スクリーンショット 2026-04-13 085158.png>)

<br>

ネットワークタブにて、下記の通り入力する

| 項目 | 入力方法 |
| -- | -- |
| 仮想ネットワーク | Spokeの仮想ネットワーク |
| サブネット | Spokeの仮想ネットワーク内の作成するサブネット名 |
| パブリック IP | 任意の仮想マシン名 |
| NIC ネットワーク セキュリティ グループ | Basic |
| パブリック受信ポート | なし |

![alt text](<スクリーンショット 2026-04-13 085235.png>)

<br>

確認及び作成で、作成を押下する<br>
![alt text](<スクリーンショット 2026-04-13 085302.png>)

<br>

## Hub環境との接続

<details><summary>詳細を見る</summary><div>

作製したSpoke環境の仮想ネットワークを開き、「ピアリング」→「追加」を押下する<br>
<img width="960" height="516" alt="image" src="https://github.com/user-attachments/assets/de79cd1e-036a-46a6-ad25-9dce658cbca1" /><br><br>

リモート仮想ネットワークの概要を以下の通り入力する<br>
※　相手先(Hub環境側)からのピアリング設定です。<br>

| 項目 | 入力方法 |
| -- | -- |
| ピアリング リンクの名前 | ピアリング名<br>例：peer-hub-prod-toSpoke1-01 |
| サブスクリプション | 相手先(Hub環境)のサブスクリプション |
| 仮想ネットワーク | 相手先(Hub環境)の仮想ネットワーク名 |

<br>

リモート仮想ネットワーク ピアリングの設定を以下の通りとする

| 項目 | 入力方法 |
| -- | -- |
| 'vnet-hub-prod-network-01' に 'vnet-spoke-prod-network-01' へのアクセスを許可する | 〇 |
| 'vnet-spoke-prod-network-01' からのトラフィック転送の受信を 'vnet-hub-prod-network-01' に許可する' | × |
| 'vnet-hub-prod-network-01' 内のゲートウェイまたはルート サーバーに 'vnet-spoke-prod-network-01' へのトラフィックの転送を許可する | 〇 |
| 'vnet-spoke-prod-network-01' のリモート ゲートウェイまたはルート サーバーを使用するために 'vnet-hub-prod-network-01' を有効にする | × |

<br>
ローカル仮想ネットワークの概要<br>
※　Spoke環境から見た設定です

| 項目 | 入力方法 |
| -- | -- |
| ピアリング リンクの名前 | ピアリング名<br>例：peer-Spoke1-prod-toHub-01 |

<br>

ローカル仮想ネットワーク ピアリングの設定

| 項目 | 入力方法 |
| -- | -- |
| 'vnet-spoke1-prod-network-01' に 'vnet-hub-prod-network-01' へのアクセスを許可する | 〇 |
| 'vnet-hub-prod-network-01' からのトラフィック転送の受信を 'vnet-spoke-prod-network-01' に許可する | × |
| 'vnet-spoke-prod-network-01' 内のゲートウェイまたはルート サーバーに 'vnet-hub-prod-network-01' へのトラフィックの転送を許可する | × |
| 'vnet-hub-prod-network-01' のリモート ゲートウェイまたはルート サーバーを使用するために 'vnet-spoke-prod-network-01' を有効にする | 〇 |

### ルートテーブルにデフォルトルート(HubのAzureFirewall)の作製
すべての通信をHubのAzureFirewallへ経由させるためのルートを作成します。

なぜAzureFirewallをネクストホップとするのか<br>
AzureFirewallは、許可した通信のみを通して、それ以外は通さない仕組みになっています。<br>
Hubで通信可否を中央集中的に管理する場合、Spoke環境の通信経路はAzureFirewallを経由させることで意図しない通信が発生しにくくなります。<br>

<details><summary>詳細を見る</summary><div>

作製したルートテーブルにて、「設定」→「ルート」より、「追加」を押下し、以下の通り作成する<br>
※　ネクストホップをHub環境のAzureFirewallに指定します

| 項目 | 入力方法 |
| -- | -- |
| ルート名 | default |
| 宛先の種類 | IP アドレス |
| ネクスト ホップの種類 | 仮想アプライアンス |
| Propagate gateway routes | Yes |
| ネクスト ホップ アドレス | Azure FirewallのプライベートIPアドレス |

![alt text](<スクリーンショット 2026-04-13 090721.png>)

<br>

### Spoke環境のサブネットにルートテーブルを関連付け
ルートテーブルは、作成するだけでは動作しません。サブネットに関連付けることでそのサブネット内のルートに反映されます。

作製したルートテーブルにて、「設定」→「サブネット」より、「関連付け」を押下し、Spoke環境のVMがあるサブネットを選択してOKする

![alt text](<スクリーンショット 2026-04-13 090853.png>)
